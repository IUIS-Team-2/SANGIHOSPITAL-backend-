BACKEND BUG
Payment mode changes for OLD admission when new admission is created
Root cause — get_preferred_admission_for_patient() in patients/serializers.py

When the frontend loads a patient, it reads patient.current_admission_detail (not the full admissions array). This field is populated by get_current_admission_detail() in PatientSerializer, which calls get_preferred_admission_for_patient().

That function picks the latest active admission (highest timestamp / admNo). So after you create Admission 2 (cash), it becomes the "preferred" admission. Every dashboard now shows Admission 2's billing — cash — even when the receptionist is looking at Admission 1's record.

Additionally, AdmissionSerializer.get_billing() does obj.bills.order_by('-id').first() — this always returns the most recent billing record, not the one for the specific admission.
Fix — backend/patients/serializers.py
patients/serializers.py — get_preferred_admission_for_patient()
The function is correct — it should return the latest active admission. The real problem is dashboards using current_admission_detail when they should be showing per-admission data. The fix is to ensure each AdmissionSerializer returns its own billing, not a cross-admission one. The billing serializer is already correct. The actual issue is the frontend reads patient.current_admission_detail.billing.bill_type to determine the payment display for the whole patient — this should instead read from the specific admission being viewed.
# The billing is correctly scoped per admission in AdmissionSerializer: def get_billing(self, obj): billing = obj.bills.order_by('-id').first() # ← this is correct — per admission # HOWEVER — the patient-level cashless_records filter is wrong: # In PatientViewSet.cashless_records and get_queryset for HOD: Patient.objects.filter(admissions__bills__bill_type='CASHLESS').distinct() # This returns a patient if ANY admission is cashless — even if current admission is cash. # This is what causes the confusion in dashboards. # FIX: Add an annotation so the frontend knows which admissions are cashless: # In AdmissionSerializer.to_representation(), add: def to_representation(self, instance): data = super().to_representation(instance) billing = instance.bills.order_by('-id').first() data['bill_type'] = billing.bill_type if billing else 'CASH' # ... rest of method return data
After this, each admission in patient.admissions[] will have its own bill_type field. Tell your frontend dev: when showing billing details, always read from the specific admission object the user is currently viewing — not from current_admission_detail which always points to the latest one.
BACKEND + FRONTEND
Why "LXM" / Lakshmi Nagar is always selected as default branch
Backend side:
In patients/models.py, the Patient model has:
branch_location = models.CharField(max_length=10, default='LNM')
And Patient.save() calls get_default_branch_code() which does:
HospitalSettings.objects.order_by('id').first()
This returns whichever branch was created first in the DB. If Lakshmi Nagar was the first branch created, it will always be the default.

Frontend side:
In SuperAdminDashboard.jsx:
const defaultBranchSlug = branches[0]?.slug || "laxmi"
The frontend takes the first branch returned by /api/hospital-settings/ as the default. The hospital-settings endpoint returns branches ordered by DB insertion order (no explicit ordering). Since Lakshmi Nagar was first created, it comes back first → branches[0] = LXM.
Fix — Backend only (frontend will auto-fix once backend sends correct order)
patients/models.py — get_default_branch_code()
# Current (uses DB insertion order): def get_default_branch_code(): default_branch = HospitalSettings.objects.order_by('id').first() return default_branch.branch if default_branch else 'LNM' # This is expected behaviour — whoever is logged in, THEIR branch should be the default. # The real fix is: don't use a model-level default at all. # In Patient.save(), the branch is always set from the request data in PatientViewSet.create(). # The default only matters if no branch is sent — which means the frontend isn't sending it.
backend/patients/views.py — PatientViewSet.create()
# In create(), after resolving branch, add this fallback: def create(self, request, *args, **kwargs): data = request.data.copy() # If no branch sent, use the logged-in user's branch (not DB first branch) if not data.get('branch_location') and not data.get('locId'): if getattr(request.user, 'branch', None) not in [None, 'ALL']: data['branch_location'] = request.user.branch # ... rest of create unchanged
This ensures that when a receptionist registers a patient and the frontend forgets to send the branch, it defaults to the receptionist's own branch — not whatever was first in the DB. If the receptionist is from RYM, the patient gets RYM. No more LXM-as-universal-default.
